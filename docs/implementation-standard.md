# AI-RPG Engine 实现标准文档

本文档整合了项目设计文档、架构决策、开发日志等内容，定义了统一的实现标准和流程规范。

---

## 一、项目概述

### 1.1 项目定位

**项目名称**: AI-RPG Engine  
**项目性质**: 免费开源项目 (MIT License)  
**核心价值**: 基于大语言模型(LLM)的通用RPG游戏框架，支持AI动态生成故事内容

### 1.2 技术栈

| 层级 | 技术选型 |
|------|----------|
| 前端框架 | React 18 + TypeScript + Vite 5 |
| 状态管理 | Zustand |
| 后端框架 | Node.js + Express + TypeScript |
| 数据库 | SQLite (sql.js) |
| AI接口 | 统一适配器模式 (DeepSeek, GLM, Kimi, OpenAI) |
| 项目结构 | Monorepo + pnpm workspace |

### 1.3 项目结构

项目采用Monorepo结构，包含三个主要包：

- **frontend**: React前端应用，包含UI组件、API服务、状态管理
- **backend**: Express后端服务，包含智能体、工具、服务、路由
- **shared**: 共享类型定义

数据存储在项目目录下的 `game_data/` 文件夹中，包含SQLite数据库和设置文件。

## 二、核心架构设计

### 2.1 架构模式：总分总 + Tool/Agent分离

采用**新架构模式**：Agent专注AI决策，Tool专注数据操作，CoordinatorAgent负责消息路由和协调。

**架构层级（自上而下）**：

1. **前端层**: React UI、GameStore状态管理、API客户端、WebSocket通信
2. **路由层**: Express路由、WebSocket服务
3. **智能体层**: CoordinatorAgent（组长）+ 11个专业智能体（组员），各司其职
4. **工具层**: 11个数据工具，提供统一的数据访问接口
5. **服务层**: 各领域业务服务
6. **基础设施层**: LLM服务、上下文管理、决策日志、数据库服务
7. **数据存储层**: SQLite数据库、游戏数据文件、提示词配置

### 2.2 数据流架构

**玩家输入处理流程**：

1. 玩家输入发送到后端
2. CoordinatorAgent接收请求，根据Binding配置进行意图分析和消息路由
3. CoordinatorAgent调度目标Agent处理消息（第一层调度）
4. 目标Agent处理消息，可能涉及：
   - 调用Tool获取或修改数据
   - 调用LLM进行AI推理
   - 如需其他Agent协作，返回需求给CoordinatorAgent（不能直接调用其他Agent）
5. CoordinatorAgent整合第一层结果，判断是否需要二次调度
6. 如需二次调度，CoordinatorAgent再次调度Agent处理（第二层调度，必须返回数据）
7. CoordinatorAgent整合所有结果，如需动态UI则调度UIAgent（附属任务）
8. CoordinatorAgent返回最终响应给前端
9. 前端更新界面显示

**调度层级示意**：

```
玩家输入
    │
    ▼
CoordinatorAgent（第一层调度）
    │
    ├──▶ Agent A ──▶ 返回结果A
    │
    ├──▶ Agent B ──▶ 返回结果B
    │
    ▼
CoordinatorAgent整合结果A+B
    │
    ├──▶ 判断需要二次调度？
    │         │
    │         ├── 是 ──▶ CoordinatorAgent（第二层调度）
    │         │              │
    │         │              ├──▶ Agent C ──▶ 返回结果C（必须返回数据）
    │         │              │
    │         │              ▼
    │         │         整合结果A+B+C
    │         │
    │         └── 否 ──▶ 继续
    │
    ├──▶ 需要动态UI？
    │         │
    │         ├── 是 ──▶ UIAgent（附属任务）──▶ 动态UI数据
    │         │
    │         └── 否 ──▶ 继续
    │
    ▼
返回最终响应给前端
```

## 三、智能体系统设计

### 3.1 Agent架构原则

**组长+组员模式**：
- **CoordinatorAgent (组长)**: 负责统筹协调、结果整合、智能调度
- **其他Agent (组员)**: 各自负责专业领域，都有LLM调用能力
- **Tool (工具)**: 专注数据读写和计算，无LLM能力

### 3.2 Agent列表与职责

| Agent | 类型标识 | 职责 | AI调用时机 |
|-------|----------|------|------------|
| CoordinatorAgent | coordinator | 统筹协调、意图分析、消息路由、结果整合、智能调度 | 意图分析、消息路由、结果整合 |
| DialogueAgent | dialogue | NPC对话生成、选项生成、情绪分析 | 对话生成、选项生成 |
| CombatAgent | combat | 战斗流程管理、AI决策、战斗叙事 | 困难模式AI决策、战斗叙事 |
| QuestAgent | quest | 任务生成、目标设计、奖励设计 | 任务生成、描述生成 |
| StoryContextAgent | story_context | 故事上下文管理、剧情维护、上下文压缩 | 上下文压缩、剧情生成 |
| MapAgent | map | 地图管理、位置处理、区域生成 | 区域描述生成 |
| NPCAgent | npc_party | NPC管理、关系系统、队伍管理 | NPC行为生成 |
| InventoryAgent | inventory | 物品管理、装备系统 | 物品生成（属性、效果、描述） |
| SkillAgent | skill | 技能管理、冷却计时 | 技能生成（效果、消耗、描述） |
| NumericalAgent | numerical | 属性计算、伤害公式、等级成长 | 数值建议、平衡分析 |
| UIAgent | ui | 动态UI生成 | 动态UI Markdown生成 |
| EventAgent | event | 事件管理、条件检测 | 事件生成、条件设计 |

### 3.3 Agent核心能力

**所有Agent共同能力**：

- **Tool调用**: 通过ToolRegistry访问数据工具（每个Agent都有此能力）
- **LLM调用**: 独立调用大语言模型进行推理
- **消息处理**: 实现processMessage方法处理接收到的消息
- **数据索引**: 对大量数据建立索引，支持快速检索和分页获取
- **分批处理**: 将大数据集分批处理，避免单次数据过长导致LLM截断

**CoordinatorAgent独有能力**：

- **意图分析**: 分析玩家输入，确定需要调用的Agent
- **消息路由**: 根据Binding配置决定消息路由
- **Agent调度**: 调度其他Agent协作处理（组员Agent不能直接调用其他Agent）
- **智能调度**: 根据依赖关系决定顺序或并行执行
- **结果整合**: 等待并整合各Agent的执行结果
- **二次调度**: 根据整合结果再次分配任务（最多2层调度）

**调度层级限制**：

- **第一层**: CoordinatorAgent接收玩家输入，调度目标Agent处理
- **第二层**: CoordinatorAgent根据第一层结果，可再次调度Agent（第二层必须返回数据）
- **附属任务**: 调度UIAgent生成动态UI不算独立层级，是返回数据的附属任务
- **禁止三层**: 不允许第三层调度，避免无限循环

**组员Agent限制**：

- 不能直接调用其他Agent
- 需要其他Agent协作时，必须返回需求给CoordinatorAgent，由CoordinatorAgent统一调度

### 3.4 Agent上下文隔离

**设计原则**：每个Agent都有独立的上下文空间，实现上下文隔离。

**隔离机制**：

- **独立上下文**: 每个Agent维护自己的上下文数据，包括对话历史、状态信息、工作记忆
- **数据隔离**: Agent之间不能直接访问彼此的上下文，避免记忆混乱
- **通信规范**: Agent间需要共享信息时，必须通过CoordinatorAgent传递，不能直接读取其他Agent的上下文
- **上下文压缩**: 各Agent负责自己上下文的压缩和管理，StoryContextAgent负责全局故事上下文

**上下文类型**：

| 上下文类型 | 管理者 | 内容 |
|------------|--------|------|
| 全局故事上下文 | StoryContextAgent | 主线剧情、世界观、重要事件 |
| 对话上下文 | DialogueAgent | 当前对话历史、NPC情绪、话题 |
| 战斗上下文 | CombatAgent | 战斗状态、敌人信息、回合记录 |
| 任务上下文 | QuestAgent | 任务进度、目标状态、奖励信息 |
| 地图上下文 | MapAgent | 当前位置、已探索区域、地点信息 |
| NPC上下文 | NPCAgent | NPC状态、关系值、行为记录 |

**上下文同步**：

- CoordinatorAgent负责协调需要跨Agent共享的信息
- 共享信息通过消息传递，而非直接访问上下文
- 避免上下文污染和记忆混乱

**按存档隔离上下文**：

- 每个存档对应独立的Agent上下文空间
- 不同存档的Agent上下文完全隔离，互不影响
- 存档保存时，必须保存所有Agent的完整上下文
- 存档读取时，必须完整恢复所有Agent的上下文
- 上下文隔离确保多存档游戏的独立性

**上下文持久化**：

| 持久化时机 | 操作 |
|------------|------|
| 存档保存 | 将所有Agent上下文序列化并写入存档 |
| 存档读取 | 从存档反序列化并恢复所有Agent上下文 |
| 自动存档 | 定期保存上下文快照 |
| 游戏退出 | 提示保存，保存当前上下文状态 |

### 3.5 Agent间通信协议

消息结构包含以下字段：

- **id**: 消息唯一标识
- **timestamp**: 时间戳
- **from**: 发送者Agent类型
- **to**: 接收者Agent类型（单个或多个）
- **type**: 消息类型（请求/响应/通知/错误）
- **payload**: 消息内容，包含动作和数据
- **metadata**: 元数据，包含优先级、是否需要响应、超时时间
- **correlationId**: 关联消息ID，用于追踪请求-响应链

### 3.6 Agent提示词工程指导

**提示词设计原则**：

| 原则 | 描述 | 示例 |
|------|------|------|
| 角色明确 | 清晰定义Agent的角色和职责 | "你是一个专业的对话生成Agent，负责生成NPC对话和玩家选项" |
| 任务聚焦 | 每个提示词只关注一个核心任务 | 避免在一个提示词中混合对话生成和战斗计算 |
| 输出规范 | 明确指定输出格式和结构 | "输出必须包含：对话内容、情绪状态、选项列表" |
| 约束清晰 | 设定明确的边界和限制 | "对话长度不超过200字，选项数量2-4个" |

**提示词结构模板**：

每个Agent的提示词应包含以下部分：

1. **角色定义**: 定义Agent的身份和职责范围
2. **任务描述**: 详细说明需要完成的任务
3. **输入说明**: 描述预期的输入内容和格式
4. **输出规范**: 明确输出的格式、字段、约束
5. **上下文注入点**: 预留动态上下文注入的位置
6. **示例参考**: 提供输入输出示例（可选）
7. **约束条件**: 列出禁止行为和限制条件
8. **Tool能力说明**: 描述可用的Tool及其功能
9. **其他智能体信息**: 说明相关Agent及其职责
10. **任务边界**: 明确任务范围和边界条件
11. **错误处理**: 定义错误情况和处理方式
12. **协作指导**: 说明与其他Agent的协作方式

**Tool能力说明**：

提示词中应明确说明Agent可使用的Tool能力：

| 说明项 | 描述 | 示例 |
|--------|------|------|
| 可用Tool列表 | 列出Agent可调用的Tool | "你可以使用DialogueDataTool和StoryDataTool" |
| Tool功能描述 | 简要说明每个Tool的功能 | "DialogueDataTool用于管理对话数据" |
| 调用权限 | 说明读写权限限制 | "你对DialogueDataTool有读写权限，对其他Tool只有读权限" |
| 调用时机 | 说明何时应该调用Tool | "当需要获取对话历史时，调用DialogueDataTool.getHistory" |

**其他智能体信息**：

提示词中应包含相关Agent的信息：

| 说明项 | 描述 | 示例 |
|--------|------|------|
| 协作Agent | 列出可能协作的Agent | "你可能需要与StoryContextAgent协作" |
| Agent职责 | 说明其他Agent的职责 | "StoryContextAgent负责维护全局故事上下文" |
| 协作方式 | 说明如何协作 | "如需全局故事信息，返回需求给CoordinatorAgent" |
| 禁止行为 | 明确不能直接调用其他Agent | "你不能直接调用其他Agent，必须通过CoordinatorAgent协调。CoordinatorAgent仅有一次协调机会，尽可能描述你所有需求" |

**任务边界**：

提示词中应明确定义任务边界：

| 边界类型 | 描述 | 示例 |
|----------|------|------|
| 职责边界 | 明确Agent负责和不负责的内容 | "你只负责对话生成，不负责战斗计算" |
| 数据边界 | 明确可访问的数据范围 | "你只能访问对话相关的数据" |
| 决策边界 | 明确决策权限 | "你可以决定对话内容，但不能修改玩家属性" |
| 触发边界 | 明确任务触发条件 | "仅在收到dialogue_request消息时触发" |

**错误处理**：

提示词中应定义错误处理策略：

| 错误类型 | 处理方式 | 示例 |
|----------|----------|------|
| 输入错误 | 返回错误信息，请求修正 | "输入缺少必要字段，请补充角色名称" |
| Tool调用失败 | 返回错误信息，提供备选方案 | "获取对话历史失败，使用默认开场白" |
| 数据缺失 | 使用默认值或请求补充 | "NPC信息缺失，使用通用NPC模板" |
| 权限不足 | 返回错误信息，请求协调 | "无权访问该数据，请CoordinatorAgent协调" |
| 超时处理 | 返回部分结果，标记未完成 | "处理超时，返回已生成的部分对话" |

**协作指导**：

提示词中应包含协作指导：

| 协作场景 | 指导内容 | 示例 |
|----------|----------|------|
| 数据共享 | 说明如何请求其他Agent的数据 | "如需任务信息，在返回中添加needData字段" |
| 任务委托 | 说明如何委托任务给其他Agent | "如需生成战斗描述，返回needAgent字段" |
| 结果整合 | 说明如何整合协作结果 | "整合StoryContextAgent提供的背景信息" |
| 冲突处理 | 说明如何处理协作冲突 | "如与其他Agent结果冲突，优先保留核心任务输出" |
| 优先级 | 说明协作任务的优先级 | "对话生成优先级高于背景描述" |

**提示词优化技巧**：

| 技巧 | 描述 | 效果 |
|------|------|------|
| 分步引导 | 将复杂任务分解为多个步骤 | 提高准确性，减少遗漏 |
| 格式标记 | 使用特殊标记区分不同内容 | 便于解析，减少格式错误 |
| 负面约束 | 明确说明不应该做什么 | 减少幻觉和错误输出 |
| 角色扮演 | 让AI扮演特定角色 | 增强一致性，提高沉浸感 |
| 思维链 | 引导AI展示推理过程 | 提高复杂任务的准确性 |

**上下文注入策略**：

| 注入类型 | 描述 | 适用场景 |
|----------|------|----------|
| 模板注入 | 预设固定的上下文模板 | 世界观、角色设定 |
| 动态注入 | 运行时动态获取上下文 | 当前状态、玩家行为 |
| 摘要注入 | 注入压缩后的上下文摘要 | 历史记录、长对话 |
| 索引注入 | 注入数据索引而非完整数据 | 大量数据场景 |

**提示词版本管理**：

- 每个Agent的提示词应有版本号
- 修改提示词需记录变更原因和效果
- 保留历史版本以便回滚和对比
- 提示词存储在数据库中，支持热更新

**提示词测试与调优**：

1. **单元测试**: 对每个Agent的提示词进行独立测试
2. **集成测试**: 测试多Agent协作时的提示词效果
3. **A/B测试**: 对比不同提示词版本的效果
4. **效果评估**: 建立评估指标（准确性、一致性、响应时间）
5. **迭代优化**: 根据测试结果持续优化提示词

## 四、Tool层设计

### 4.1 Tool架构原则

- **职责单一**: 每个Tool专注一个领域的数据操作
- **读写分离**: 明确区分读方法和写方法
- **无LLM能力**: Tool仅负责数据读写和计算
- **统一接口**: 所有Tool继承ToolBase基类

### 4.2 Tool列表

| Tool | 类型标识 | 职责 | 主要方法 |
|------|----------|------|----------|
| InventoryDataTool | inventory_data | 背包数据管理 | 获取物品、添加物品、移除物品、使用物品 |
| SkillDataTool | skill_data | 技能数据管理 | 获取技能、学习技能、升级技能、使用技能 |
| MapDataTool | map_data | 地图数据管理 | 获取位置、移动角色、探索区域 |
| QuestDataTool | quest_data | 任务数据管理 | 获取任务、更新进度、完成任务 |
| NPCPartyDataTool | npc_data | NPC和队伍管理 | 获取NPC、更新关系、获取队伍 |
| EventDataTool | event_data | 事件管理 | 获取事件、触发事件、解决事件 |
| StoryDataTool | story_data | 剧情管理 | 获取故事上下文、更新故事、获取历史 |
| UIDataTool | ui_data | UI指令管理 | 生成指令、格式化文本 |
| DialogueDataTool | dialogue_data | 对话数据管理 | 获取对话、添加消息、获取历史 |
| CombatDataTool | combat_data | 战斗数据管理 | 获取战斗状态、执行行动、计算伤害 |
| NumericalTool | numerical | 数值计算 | 计算属性、计算伤害、计算经验 |

### 4.3 Tool核心能力

每个Tool具备以下核心能力：

- **方法注册**: 注册读方法和写方法
- **权限控制**: 区分读方法和写方法的权限
- **执行入口**: 统一的execute方法
- **响应创建**: 创建成功或错误响应
- **状态管理**: 获取运行状态

### 4.4 Tool权限控制机制

**权限分级**：

| 权限级别 | 描述 | 适用对象 |
|----------|------|----------|
| 全部读写 | 可调用所有Tool的所有方法 | CoordinatorAgent |
| 职责内读写 | 可读写职责内Tool，只读其他Tool | 各专业Agent |
| 只读 | 只能调用读方法 | 特殊场景 |

**权限分配规则**：

- **CoordinatorAgent**: 拥有所有Tool的全部读写权限
- **专业Agent**: 
  - 对职责内Tool拥有读写权限
  - 对其他Tool只有读权限
  - 执行写操作时，必须在返回数据中记录写操作内容

**职责内Tool对应关系**：

| Agent | 职责内Tool（读写权限） | 其他Tool（只读权限） |
|-------|------------------------|----------------------|
| DialogueAgent | DialogueDataTool | 其他所有Tool |
| CombatAgent | CombatDataTool | 其他所有Tool |
| QuestAgent | QuestDataTool | 其他所有Tool |
| MapAgent | MapDataTool | 其他所有Tool |
| NPCAgent | NPCPartyDataTool | 其他所有Tool |
| InventoryAgent | InventoryDataTool | 其他所有Tool |
| SkillAgent | SkillDataTool | 其他所有Tool |
| StoryContextAgent | StoryDataTool | 其他所有Tool |
| EventAgent | EventDataTool | 其他所有Tool |
| NumericalAgent | NumericalTool | 其他所有Tool |
| UIAgent | UIDataTool | 其他所有Tool |

**写操作记录机制**：

当Agent执行写操作时，必须遵循以下流程：

1. Agent调用Tool的写方法
2. Tool执行写操作并返回结果
3. Agent在返回给CoordinatorAgent的数据中记录写操作内容：
   - 写操作的Tool类型
   - 写操作的方法名
   - 写操作的参数
   - 写操作的结果
4. CoordinatorAgent整合所有写操作记录，防止二次写入冲突

**写操作记录结构**：

| 字段 | 说明 |
|------|------|
| toolType | Tool类型标识 |
| method | 写方法名称 |
| params | 写操作参数 |
| result | 写操作结果 |
| timestamp | 操作时间戳 |

**防二次写入机制**：

- CoordinatorAgent维护一个写操作日志
- 每次整合结果时，检查是否有重复的写操作
- 发现重复写操作时，跳过后续执行并记录警告
- 确保同一数据不会被多次修改

### 4.5 数据索引与分批处理机制

**设计目的**：防止数据过长导致LLM截断，确保Agent能够高效处理大量数据。

**数据索引机制**：

| 索引类型 | 描述 | 应用场景 |
|----------|------|----------|
| 关键词索引 | 按关键词建立索引 | 快速检索相关内容 |
| 时间索引 | 按时间顺序建立索引 | 历史记录查询 |
| 类型索引 | 按数据类型建立索引 | 分类数据检索 |
| 关联索引 | 按关联关系建立索引 | 关联数据查询 |

**分批处理策略**：

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| 固定大小分批 | 按固定条数分批 | 列表数据 |
| Token限制分批 | 按Token数量分批 | 文本数据 |
| 优先级分批 | 按重要性分批，优先处理重要数据 | 任务、事件 |
| 增量分批 | 只处理变化的数据 | 状态更新 |

**分批处理流程**：

1. Agent接收数据处理请求
2. 评估数据总量和Token预算
3. 根据策略确定分批方案
4. 按批次处理数据
5. 汇总各批次结果
6. 返回最终结果

**Token预算管理**：

- 每个Agent有预设的Token预算上限
- 数据分批时需预留上下文和响应空间
- 超出预算时自动触发分批处理
- 优先保留关键信息，次要信息可摘要或省略

**数据摘要机制**：

当数据量过大时，可采用摘要策略：

- **压缩摘要**: 保留核心信息，压缩冗余内容
- **抽样摘要**: 选取代表性数据作为样本
- **层级摘要**: 提供概览，按需展开详情
- **时间窗口**: 只处理最近N条或N时间内的数据

## 七、角色创建流程

### 7.1 创建流程

```
主菜单 → 模板选择 → 角色创建 → 游戏界面
                     ├── Step 1: 名称输入
                     ├── Step 2: 种族选择
                     ├── Step 3: 职业选择
                     ├── Step 4: 背景选择
                     └── Step 5: 角色确认
```

### 7.2 选项生成策略

**模式**: 模板+LLM生成，玩家可选

- **模板驱动**: 从模板读取预定义的种族/职业/背景选项
- **LLM增强**: 设置中"AI随机生成角色选项"开启时，发送需求给CoordinatorAgent，由其调用LLM额外生成种族/职业/背景各3个选项
- **一致性保证**: 所有LLM调用通过CoordinatorAgent统一管理，不直接调用LLM
- **一次调用**: 每个种族/职业/背景选项仅调用一次LLM，结果缓存后重复使用

### 7.3 属性计算

**计算公式**：
```
最终属性 = 基础值 + 种族加成/惩罚 + 职业加成 + 背景加成
```

## 八、战斗系统设计

### 8.1 混合战斗模式

**切换逻辑**: 根据模板配置的战斗类型自动选择

| 战斗类型 | 描述 | 适用场景 |
|----------|------|----------|
| 叙事型 | AI生成战斗描述，数值系统辅助 | 文字冒险、视觉小说 |
| 回合制 | 严格回合制，完整数值计算 | 回合制RPG |
| 混合型 | 根据场景动态切换 | 动态战斗 |

### 8.2 回合制战斗流程

```
战斗初始化
    │
    ▼
计算先攻顺序（基于速度属性）
    │
    ▼
玩家回合
    ├── 攻击
    ├── 技能
    ├── 物品
    ├── 防御
    └── 逃跑
    │
    ▼
敌人回合
    ├── 简单AI（普通模式）
    ├── 规则AI（困难模式）
    └── LLM决策（困难模式）
    │
    ▼
检查战斗结束
    ├── 继续 → 返回玩家回合
    └── 结束 → 发放奖励
```

### 8.3 伤害计算公式

**物理伤害**：
- 基础伤害 = 攻击力 × 2 - 防御力
- 随机系数 = 0.9 ~ 1.1
- 暴击倍率 = 1.5（默认）
- 最终伤害 = 基础伤害 × 随机系数 × 暴击倍率

**魔法伤害**：
- 基础伤害 = 技能基础伤害
- 智力系数加成
- 抗性减免

**暴击判定**：
- 暴击率 = 基础暴击率 + 幸运值/100

## 十、游戏初始化流程

### 10.1 初始化原则

**核心原则**：

1. **模板驱动**: 所有初始化必须严格按照游戏模板的要求进行
2. **单次调用**: 角色创建完成后，仅调用一次CoordinatorAgent，由其统一调度所有初始化工作
3. **玩家选择驱动**: 根据玩家选择的种族、职业、背景进行个性化初始化
4. **数据一致性**: 游戏运行必须严格按照初始化产生的数据工作，不得随意偏离
5. **欢迎体验**: 初始化完成后必须通过动态UI提供欢迎信息，引导玩家进入游戏

### 10.2 初始化流程

```
角色创建完成
    │
    ▼
CoordinatorAgent接收初始化请求（仅调用一次）
    │
    ├── 第一阶段：基础数据初始化（并行执行）
    │    │
    │    ├── 1. NumericalAgent.calculateInitialStats()
    │    │      - 根据模板数值规则计算基础属性
    │    │      - 应用玩家选择的种族加成/惩罚
    │    │      - 应用玩家选择的职业加成
    │    │      - 应用玩家选择的背景加成
    │    │
    │    ├── 2. SkillAgent.initializeSkills()
    │    │      - 根据模板技能规则添加初始技能
    │    │      - 根据玩家职业添加职业初始技能
    │    │      - 根据玩家背景添加背景特殊技能
    │    │
    │    ├── 3. InventoryAgent.initializeInventory()
    │    │      - 根据模板物品规则添加初始物品
    │    │      - 根据玩家职业添加职业初始物品
    │    │      - 根据玩家背景添加背景初始物品
    │    │      - 添加模板定义的初始金币
    │    │
    │    └── 4. MapAgent.initializeStartingLocation()
    │           - 根据模板定义的起始场景设置初始位置
    │           - 初始化模板定义的可探索区域
    │
    ├── 第二阶段：依赖数据初始化（顺序执行）
    │    │
    │    ├── 5. InventoryAgent.equipInitialGear()
    │    │      - 根据玩家职业自动装备初始装备
    │    │      - 更新装备带来的属性加成
    │    │
    │    ├── 6. QuestAgent.createMainQuest()
    │    │      - 根据模板创建主线任务
    │    │      - 设置模板定义的任务目标
    │    │
    │    └── 7. NPCAgent.initializeSceneNPCs()
    │           - 根据模板加载起始场景的NPC
    │           - 初始化NPC关系（根据玩家背景调整）
    │
    ├── 第三阶段：场景生成（顺序执行）
    │    │
    │    └── 8. DialogueAgent.generateInitialScene()
    │           - 根据模板世界观生成故事背景介绍
    │           - 根据玩家选择生成角色登场描述
    │           - 根据模板起始场景生成当前处境说明
    │           - 生成2-5个符合模板风格的快速选项
    │
    └── 第四阶段：欢迎信息（最后执行）
         │
         └── 9. UIAgent.generateWelcomeUI()
                - 生成欢迎通知（:::notify组件）
                - 生成角色摘要卡片（:::character-status组件）
                - 生成初始场景描述
                - 生成快速开始选项（:::options组件）
```

### 10.3 智能调度

CoordinatorAgent根据依赖关系自动决定顺序或并行：

- **无依赖**: 并行执行（如数值计算、技能初始化、背包初始化、地图初始化）
- **有依赖**: 顺序执行（如装备初始化依赖背包初始化，NPC初始化依赖地图初始化）
- **结果整合**: 等待所有Agent完成后整合结果
- **等待能力**: CoordinatorAgent需要有能力等待各Agent的结果并进行整合

### 10.4 数据一致性保证

**初始化数据约束**：

| 约束类型 | 描述 | 示例 |
|----------|------|------|
| 模板约束 | 所有数据必须符合模板定义的范围和规则 | 属性值不超过模板定义的最大值 |
| 选择约束 | 数据必须反映玩家的种族/职业/背景选择 | 战士职业必须获得力量加成 |
| 关联约束 | 相关数据必须保持一致 | 装备武器后攻击力必须正确更新 |
| 唯一约束 | 某些数据必须唯一 | 初始任务ID不能重复 |

**运行时数据一致性**：

- 游戏运行时必须严格按照初始化产生的数据工作
- 任何数据修改必须通过Tool进行，并记录操作日志
- 禁止绕过Tool直接修改数据
- 定期检查数据完整性

### 10.5 欢迎信息规范

**动态UI欢迎信息结构**：

1. **系统通知**: 使用 `:::notify{type=welcome}` 组件显示欢迎信息
2. **角色摘要**: 使用 `:::character-status` 组件显示角色信息
3. **场景描述**: 使用标准Markdown显示初始场景描述

### 11.3 存档一致性保证

**保存一致性**：

| 保证项 | 描述 |
|--------|------|
| 原子性 | 存档操作要么完全成功，要么完全失败，不存在部分保存 |
| 完整性 | 必须保存所有必要数据，包括智能体上下文和数据索引 |
| 一致性 | 存档时暂停所有Agent操作，确保数据处于一致状态 |
| 校验和 | 保存时计算校验和，读取时验证数据完整性 |

**读取一致性**：

| 保证项 | 描述 |
|--------|------|
| 完整恢复 | 读取存档后，所有Agent上下文和数据索引必须完整恢复 |
| 状态同步 | 所有Agent的状态必须与存档时一致 |
| 索引重建 | 如索引损坏，必须能够从原始数据重建索引 |
| 版本兼容 | 支持存档版本迁移，确保旧版本存档可读取 |

### 13.4 数据隔离原则

**存档间数据隔离**：

- 每个存档有唯一的save_id
- 所有游戏状态数据通过save_id关联
- 不同存档的数据完全隔离，互不影响
- 智能体上下文按存档隔离存储

**运行时与持久化分离**：

- 运行时数据存储在agent_contexts等表中（当前游戏）
- 持久化数据通过存档系统保存到save_*表中
- 读取存档时，将save_*表数据恢复到运行时表

## 十八、用户确认的关键设计决策

### 18.1 Agent架构选择

**选择**: 新架构 (Tool+Binding)

**说明**: 
- Agent专注AI决策，通过Tool访问数据
- Tool专注数据操作，无LLM能力
- Binding配置定义消息路由规则，由CoordinatorAgent读取和使用
- CoordinatorAgent负责消息路由和协调，无需独立的BindingRouter
- 架构清晰，职责分离

### 18.2 AI调用策略

**选择**: 最大化调用 + 组长+组员模式

**说明**:
- 所有Agent都有LLM调用能力
- 所有Agent都有Tool调用能力（访问数据）
- CoordinatorAgent作为组长，负责统筹协调、消息路由、Agent调度
- 其他Agent作为组员，各自负责专业领域
- 组员Agent不能直接调用其他Agent，需要协作时必须返回需求给CoordinatorAgent
- Tool作为工具，仅提供数据读写和计算能力，弥补LLM数据计算能力弱的缺陷

### 18.3 调度层级限制

**选择**: 最多2层调度

**说明**:
- 第一层：CoordinatorAgent接收玩家输入，调度目标Agent处理
- 第二层：CoordinatorAgent根据第一层结果，可再次调度Agent（第二层必须返回数据）
- 附属任务：调度UIAgent生成动态UI不算独立层级，是返回数据的附属任务
- 禁止三层：不允许第三层调度，避免无限循环

### 18.4 Agent上下文隔离

**选择**: 独立上下文，隔离管理

**说明**:
- 每个Agent维护独立的上下文空间，包括对话历史、状态信息、工作记忆
- Agent之间不能直接访问彼此的上下文，避免记忆混乱
- 需要共享信息时，必须通过CoordinatorAgent传递
- 各Agent负责自己上下文的压缩和管理
- StoryContextAgent负责全局故事上下文的维护

### 18.5 Tool权限控制

**选择**: 分级权限机制

**说明**:
- CoordinatorAgent拥有所有Tool的全部读写权限
- 专业Agent对职责内Tool拥有读写权限，对其他Tool只有读权限
- 例如：DialogueAgent可读写DialogueDataTool，但只能读取InventoryDataTool
- 权限隔离防止数据被意外修改

### 18.6 写操作记录

**选择**: 必须记录写操作

**说明**:
- Agent执行写操作时，必须在返回给CoordinatorAgent的数据中记录写操作内容
- 记录内容包括：Tool类型、方法名、参数、结果、时间戳
- CoordinatorAgent维护写操作日志，检查重复写操作
- 发现重复写操作时跳过执行并记录警告，防止二次写入冲突

### 18.7 数据索引与分批处理

**选择**: 索引+分批处理机制

**说明**:
- 所有Agent都具备数据索引和分批处理能力
- 数据索引支持关键词、时间、类型、关联等多种索引方式
- 分批处理支持固定大小、Token限制、优先级、增量等策略
- 每个Agent有预设的Token预算上限，超出时自动触发分批
- 数据量过大时可采用压缩摘要、抽样摘要、层级摘要等策略
- 防止数据过长导致LLM截断，确保高效处理大量数据

### 18.8 提示词工程指导

**选择**: 结构化提示词设计

**说明**:
- 每个Agent的提示词应包含12个部分：角色定义、任务描述、输入说明、输出规范、上下文注入点、示例参考、约束条件、Tool能力说明、其他智能体信息、任务边界、错误处理、协作指导
- 提示词设计遵循：角色明确、任务聚焦、输出规范、约束清晰
- Tool能力说明：可用Tool列表、功能描述、调用权限、调用时机
- 其他智能体信息：协作Agent、Agent职责、协作方式、禁止行为
- 任务边界：职责边界、数据边界、决策边界、触发边界
- 错误处理：输入错误、Tool调用失败、数据缺失、权限不足、超时处理
- 协作指导：数据共享、任务委托、结果整合、冲突处理、优先级
- 优化技巧：分步引导、格式标记、负面约束、角色扮演、思维链
- 上下文注入策略：模板注入、动态注入、摘要注入、索引注入
- 提示词版本管理：版本号、变更记录、历史版本、热更新支持
- 测试与调优：单元测试、集成测试、A/B测试、效果评估、迭代优化

### 18.9 战斗模式切换

**选择**: 根据模板配置自动选择

**说明**:
- 根据模板中配置的战斗类型自动选择战斗模式
- 支持叙事型、回合制、混合型三种模式
- 无需玩家手动切换

### 18.10 UI Agent职责

**选择**: 方案B - 仅在CoordinatorAgent认为需要时调用

**说明**:
- 固定面板的数据由CoordinatorAgent整合后，直接调用UIDataTool发送到前端
- 动态UI由CoordinatorAgent调用UIAgent，UIAgent利用LLM生成Markdown动态组件，然后调用UIDataTool发送到前端
- UI Agent不主动工作，仅在CoordinatorAgent认为现有固定面板无法满足游戏交互需要时才被调用

### 18.11 角色创建流程

**选择**: 模板+LLM生成，玩家可选

**说明**:
- 从模板读取预定义的种族/职业/背景选项
- 设置中"AI随机生成角色选项"开启时，发送需求给CoordinatorAgent，由其调用LLM额外生成3个选项
- 所有LLM调用通过CoordinatorAgent统一管理，保证一致性，不直接调用LLM

### 18.12 游戏初始化流程

**选择**: 智能调度

**说明**:
- CoordinatorAgent根据依赖关系自动决定顺序或并行
- 无依赖的任务并行执行
- 有依赖的任务顺序执行
- CoordinatorAgent需要有能力等待各Agent的结果并进行整合

### 18.13 Binding配置存储

**选择**: 数据库存储

**说明**:
- Binding配置存储在数据库中
- 支持动态修改
- 前端设置面板可配置

### 18.14 存档策略

**选择**: 自动存档 + 手动存档

**说明**:
- 场景切换、任务完成时自动存档
- 玩家可随时手动存档
- 平衡便利性和玩家控制

### 18.15 开发者工具默认状态

**选择**: 默认隐藏

**说明**:
- 开发者工具默认隐藏
- 需要在设置中开启"开发者模式"才显示
- 避免干扰普通用户

### 18.16 数据存储位置

**选择**: 项目目录

**说明**:
- 数据存储在游戏目录下的game_data/文件夹
- 便于备份和便携使用
- 符合当前实现
